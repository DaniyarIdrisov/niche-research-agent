# Evaluations — методология оценки

## Зачем evals для агентной системы

Классическая ML-метрика «accuracy на test set» не подходит:
- Выход — длинный markdown-отчёт + структурированный PRD. Точечного «правильного ответа» нет.
- Качество — комбинация корректности (числа сходятся), полноты (все секции есть) и читабельности.
- Поведение зависит от температуры — два запуска того же запроса дают слегка разные тексты.

Поэтому evals у нас многоуровневые.

## Уровни оценки

### 1. Component-level

Каждый компонент тестируется изолированно.

| Компонент | Метрика | Порог |
|---|---|---|
| Scout (LLM parse) | parse_rate на 15 eval-queries | ≥ 0.95 |
| Scout | category_match (правильно угадал слаг) | ≥ 0.80 |
| Specs normalizer | accuracy на 22-парной разметке (синоним → canonical) | ≥ 0.85 |
| USP classifier (rule-based) | accuracy на 14 размеченных фразах | ≥ 0.70 |
| PRD validator | golden=valid AND broken=invalid | strict |

**Реализация:** `evals/component_evals.py`. Дамп — `last_component_results.json`.

### 2. System-level

Полный pipeline на eval-queries.

**Что проверяем на каждый запрос:**
- `task_completed` — есть финальный отчёт
- `scout_min_products` — Scout нашёл достаточно товаров
- `prd_sections_filled` — обязательные секции PRD не пустые
- `verdict_in` — verdict из допустимого множества
- `must_mention_in_report` — содержит ли отчёт ключевые слова

**Агрегаты:**
- `task_success_rate` — доля запросов, прошедших все обязательные проверки
- `avg_judge_score` — средняя оценка LLM-as-judge (1-5)
- `latency_p95` — должна быть ≤ 5 минут (см. TZ NFR)

**Реализация:** `evals/system_evals.py`. Дамп — `last_system_results.json`.

### 3. RAG metrics

Когда есть размеченные `(query, golden_chunks)` пары:
- **Precision@K** — сколько из топ-K релевантны
- **Recall@K** — сколько релевантных найдено
- **MRR** — позиция первого релевантного

В текущей реализации pairs не размечены — реализовать после первой ручной разметки на 10-20 запросах против KB.

В ноутбуке `run_evals.ipynb` секция «RAG sanity check» выводит топ-3 chunks для тестовых запросов — это быстрый визуальный smoke-test.

### 4. LLM-as-judge

Промпт — `evals/judge_prompts/report_quality.md`. Шкала 1-5:
- 5 — все секции, цифры консистентны, вердикт обоснован
- 4 — мелкие огрехи
- 3 — пропуск одной секции, общий вывод понятен
- 2 — фрагментарный отчёт
- 1 — мусор или дисквалификатор (нет verdict / галлюцинация цифр)

**Известный bias-риск:** судья и обвиняемый — одна локальная LLM (qwen2.5:3b). Для нашей задачи это приемлемо, потому что:
1. Эталона нет, какой-нибудь судья нужен.
2. Промпт судьи акцентирован на формальные критерии (есть/нет секций, формат verdict), где self-bias меньше.
3. Альтернатива — облачный judge (GPT-4) — нарушает требование локальности.

Для прода имеет смысл прогонять judge через **другую** модель — например, Llama 3.2 3B как judge, qwen 2.5 3B как агент. Это снижает self-bias.

## Эталонный набор (eval_queries.json)

15 запросов, разбитые на:
- 8 happy-path (по 2-3 на категорию)
- 2 edge case (пустая ниша, очень узкий фильтр)
- 2 с ожидаемым verdict (go vs no-go)
- 1 compliance-проверка
- 1 короткий нечёткий запрос
- 1 сравнение сегментов (out-of-scope, проверяем graceful)

Каждый запрос содержит `expected` блок с проверяемыми условиями. См. `data/eval_queries.json`.

## Пороги приёмки

Запуск считаем «успешным», если на текущем eval-наборе:

| Метрика | Порог |
|---|---|
| Component: specs_normalizer accuracy | ≥ 0.85 |
| Component: USP classifier accuracy | ≥ 0.70 |
| Component: Scout parse_rate | ≥ 0.95 |
| System: task_success_rate | ≥ 0.80 |
| System: avg judge score | ≥ 3.5 |
| System: latency_p95 | ≤ 300 sec |

Эти числа — calibration target, не гарантия. На GTX 1050 Ti latency_p95 у нас близко к границе, на ноуте без CUDA — заведомо вылетит. В этом случае выставляем `OLLAMA_LLM_MODEL=qwen2.5:1.5b-instruct-q4_K_M` (меньшая модель, выше скорость, ниже качество — баланс).

## Что отличает оценку **LLM** от оценки **агентной системы**

Классический LLM-eval (HELM, MMLU и т.д.) измеряет:
- знания модели
- логику, рассуждение
- безопасность

**Это нам не сильно интересно** — мы не оцениваем qwen2.5:3b как таковой.

Агентная система оценивается **end-to-end** — насколько pipeline решает заданную задачу. Поэтому у нас:
- **Task success rate** — главная метрика.
- **Trajectory analysis** — правильная ли последовательность вызовов (через state.json: появились ли `niche_metrics`, `specs_summary`, etc).
- **Tool calling accuracy** — для Scout: правильно ли он извлёк фильтры; для специалистов: правильно ли использовали calculator/normalizer.
- **Component fallback rate** — как часто система деградировала на fallback (важно для дебага, не для приёмки).

## Как запускать

```bash
# Компонент-уровень (быстро, ~30 сек)
uv run python -m evals.component_evals

# Систем-уровень (медленно, на 15 запросов ~10-15 мин на 1050 Ti)
uv run python -m evals.system_evals

# Только подмножество запросов
uv run python -m evals.system_evals q01_toothbrush_basic q06_epilator_basic

# Интерактивный анализ с графиками
uv run jupyter lab evals/run_evals.ipynb
```
