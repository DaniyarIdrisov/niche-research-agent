# Изоляция

## Спектр вариантов

Изоляция — про **границы доверия** между кодом, файловой системой, сетью и ядром. Чем строже изоляция, тем больше overhead и сложности.

| Способ | Изоляция | Overhead | Когда оправдано |
|---|---|---|---|
| Голый процесс | нет | 0 | trusted-код в одной среде |
| venv / conda | только зависимости Python | 0 | защита от dependency hell, не от кода |
| Docker контейнер | процесс, ФС, сеть, namespaces | низкий (~1-3%) | стандарт для микросервисов |
| Docker + gVisor | + user-space ядро | средний (~10-15%) | исполнение чужого кода |
| Firecracker microVM | полная VM, hypervisor | низкий (~3-5%) на старте, но запуск ~125 мс | serverless, function-as-a-service |
| Полная VM (KVM/VMware) | hardware-level | средний (~5-10%) | разные ОС, максимум изоляции |
| WASM (Wasmtime) | sandbox внутри процесса | очень низкий | плагины, скрипты от пользователя |

## Что выбрали для проекта

**Docker Compose** — все сервисы (ChromaDB, Jaeger, Prometheus, Grafana, Langfuse, OTel collector) запускаются в контейнерах. **Ollama** — на хосте.

**Дополнительно:** контейнер `wb-parser` (live-режим) — с усиленным режимом изоляции:
- `read_only: true` — корневая ФС read-only
- `tmpfs: /tmp` — единственная writable точка
- `security_opt: no-new-privileges:true` — нельзя поднять privileges через setuid-бинарь
- `cap_drop: ALL` — все capabilities сброшены
- собственный network: общий с приложением, но без доступа к хосту через `host` сеть

См. `docker-compose.yml`, profile `live`.

## Почему этого достаточно для университетского проекта

1. **Trust boundary:** код проекта пишет один человек, ревьюится одним человеком. Угроза «вредоносный агент» отсутствует. Изоляция нужна **от ошибок**, не **от противника**.
2. **Внешние данные минимальны.** WB API возвращает JSON, который мы валидируем через Pydantic — этого хватает. Не запускаем чужой код в проде.
3. **Ollama на хосте.** GPU прокинуть в Docker на 1050 Ti можно, но это даёт ~5-10% потерь и хрупкую настройку. Безопасность не страдает: Ollama локальный, без auth, слушает 127.0.0.1.
4. **Docker как минимальный стандарт.** Дальше gVisor/Firecracker — это уже для prod-инфры, которая обслуживает несколько арендаторов.

## Что бы добавил в проде

### 1. gVisor для исполнения LLM-сгенерированного кода

Если бы агент **выполнял** сгенерированные Python-скрипты (для кастомной аналитики), они должны были бы исполняться в gVisor-контейнере:
- Все syscalls проходят через user-space sentry → нет прямого доступа к host kernel.
- На случай RCE-эксплойта в Python — sentry становится дополнительной границей.

Реализуется флагом `runtime: runsc` в `docker-compose.yml` после установки runsc.

### 2. Firecracker для multi-tenant deployment

Если запускать систему как сервис для десятков пользователей:
- Каждый запуск — отдельный microVM (125 мс старт).
- Гарантия, что данные одного клиента не утекут другому.
- AWS Lambda работает ровно так.

Сложность в setup'е — нужен KVM на хосте, отдельный orchestrator (Firecracker-containerd или Kata).

### 3. Network policies

Сейчас контейнеры compose'а в одной сети с открытыми портами наружу. В проде:
- Закрыть все порты, кроме грейтвея.
- Внутренние сервисы (Prometheus → Grafana, OTel → Jaeger) — на private network.
- WB-парсер — egress-only, без листа портов наружу.

### 4. Secrets management

Сейчас `.env` коммитится через `.env.example` и реальный `.env` в `.gitignore`. Для прода:
- Vault или Doppler для секретов.
- Никаких ключей в файлах на диске сервера.
- Audit log на доступ.

Для университетского проекта это **out of scope** — но в защите имеет смысл упомянуть, что мы это понимаем.

### 5. WASM для плагинов

Если в будущем агент будет принимать **пользовательские плагины** (например, кастомные tools или валидаторы), запускать их в WASM-runtime (Wasmtime) даёт:
- ~zero overhead.
- Полную изоляцию: WASM не может делать syscalls без явного импорта.
- Кросс-язычность: плагин на Go, Rust, AssemblyScript — без проблем.

Сейчас не нужно, но архитектурно tools/* можно превратить в plugin-системе без переписывания.

## Сводная таблица для устной защиты

| Контур | Что защищаем | Чем |
|---|---|---|
| Зависимости | dependency hell | uv lockfile + venv |
| Сервисы | взаимная сепарация | Docker Compose |
| WB-парсер | от RCE в сторонней либе | read_only + cap_drop + no-new-privileges |
| Будущий LLM-tool | от RCE в генерируемом коде | (план) gVisor |
| Multi-tenant | от cross-tenant утечек | (план) Firecracker microVM |
| Плагины | от любого untrusted кода | (план) WASM/Wasmtime |

## Что НЕ изолировано (сознательно)

- **Ollama на хосте** — не в контейнере. Trade-off за производительность GPU.
- **`logs/` и `data/` на хост-ФС** — для удобства отладки. В проде должны быть тома с правами.
- **CLI запускается под пользователем хоста** — нет sandbox'а. В проде CLI можно завернуть в Docker.

Эти решения обоснованы: для university-проекта нужна простота развёртывания. Trade-off явный и описан.
