# Техническое задание

## 1. Цели и задачи системы

**Цель проекта:** автоматизировать первичный ресёрч ниши на маркетплейсе Wildberries в категории **малая бытовая техника (МБТ)** — от сбора данных до формирования продуктового задания (PRD) на новый товар и вердикта go/no-go.

**Задачи:**
1. Собрать топ карточек по запросу пользователя.
2. Посчитать метрики ниши (ёмкость, концентрация, ценовые сегменты, динамика спроса).
3. Извлечь и нормализовать характеристики товаров из топа.
4. Проанализировать УТП конкурентов и выявить пустоты для отстройки.
5. Сформировать PRD на новый товар по шаблону.
6. Выдать пользователю аналитический отчёт и вердикт.

**Проектные цели:**
- Защитимый университетский проект: каждое архитектурное решение задокументировано с альтернативами.
- Полностью локальный запуск на потребительском железе, без облачных API.
- Воспроизводимость: один `docker compose up` + `uv sync` поднимают всё.

---

## 2. Целевая аудитория

**Первичный пользователь:** категорийный менеджер маркетплейса / продакт-менеджер бренда МБТ на этапе принятия решения «выходить или не выходить в нишу».

**Контекст использования:**
- Пользователь знаком с экосистемой WB, понимает терминологию (СПП, оборот, БРВ, БРВ-1, ABC-XYZ).
- Доверяет агенту первичную аналитику, но финальное решение оставляет за собой.
- Запускает систему локально, не хочет отправлять стратегические запросы в облако.

**Не-цели:** система **не для конечного покупателя** и не для арбитража с другими маркетплейсами.

## 3. Функциональные требования

### 3.1. Обязательные

| Код | Требование |
|---|---|
| FR-01 | Принимать запрос на естественном языке: категория + опциональные фильтры (макс. цена, регион, бренд) |
| FR-02 | Возвращать аналитический отчёт по нише: оборот топ-30, доля топ-5 продавцов, медианная цена, спред, динамика отзывов |
| FR-03 | Возвращать PRD на новый товар по фиксированному шаблону (см. `docs/PRD_TEMPLATE.md`) |
| FR-04 | Возвращать анализ УТП топ-5 с разбивкой по типам (технологические / эмоциональные / value / социальные) и матрицей пустот |
| FR-05 | Выдавать вердикт go/no-go с обоснованием через явный набор критериев |
| FR-06 | Поддерживать два режима данных: `mock` (по умолчанию) и `live` (search.wb.ru) |
| FR-07 | Сохранять историю запусков с возможностью вернуться к предыдущему результату по run-id |
| FR-08 | Логировать промпты, ответы и трейсы каждого шага агентов |

### 3.2. Опциональные

| Код | Требование |
|---|---|
| FR-09 | Сравнение двух ниш в одном запросе |
| FR-10 | Экспорт отчёта в PDF / DOCX |
| FR-11 | Веб-UI (CLI достаточен для целей проекта) |

## 4. Нефункциональные требования

### 4.1. Производительность

| Метрика | Целевое значение | Условие |
|---|---|---|
| End-to-end latency на 1 запрос | ≤ 5 минут | GTX 1050 Ti, mock-режим, KB прогрет |
| LLM throughput | ≥ 15 токенов/сек | Qwen2.5 3B Q4_K_M на 1050 Ti |
| Время старта (cold) | ≤ 30 сек | После `docker compose up`, без загрузки моделей |

### 4.2. Аппаратные ограничения

- GPU: NVIDIA с ≥4 ГБ VRAM (минимум — 1050 Ti). Тестировалось на ней.
- RAM: 16 ГБ (на 8 ГБ — degraded mode с CPU-LLM).
- Диск: 15 ГБ под модели и образы.

### 4.3. Надёжность

- Каждый агент имеет валидацию выхода и до 2 retry при невалидном результате.
- Падение одного агента не должно ронять весь pipeline без диагностики (graceful degradation: если `Specs Miner` упал — PRD строится с пометкой «характеристики недоступны»).
- LangGraph checkpointing — можно возобновить упавший запуск с последней успешной ноды.

### 4.4. Observability

- Трейсы каждого шага в Jaeger (OpenTelemetry).
- Метрики (latency, токены, retry count, success rate) в Prometheus + Grafana.
- Промпты и ответы LLM в Langfuse.
- Структурированные JSON-логи на диск.

### 4.5. Безопасность

- Никакие данные не покидают локальную машину (кроме обращений к `search.wb.ru` в live-режиме).
- WB-парсер в live-режиме — в отдельном контейнере с `read_only`, `no-new-privileges`, `cap_drop: ALL` (см. `docs/ISOLATION.md`).
- Секретов в проекте нет — Langfuse и Grafana запускаются с dev-credentials, для прода нужны реальные.

---

## 5. Архитектурные ограничения

- **Только локальный запуск.** Никаких облачных LLM/embedding API.
- **Только OSS-модели.** Qwen2.5, Llama 3.2, nomic-embed-text, bge-reranker.
- **Только OSS-компоненты.** LangGraph, Ollama, ChromaDB, Prometheus, Grafana, Jaeger, Langfuse (self-hosted).
- **Никаких тяжёлых рантаймов.** vLLM исключён — требует >4 GB VRAM и современную CUDA.
- **Никаких тяжёлых БД.** PostgreSQL только под Langfuse, основная episodic память — SQLite.

## 6. Метрики успеха

Запуск считается успешным, если:

| Критерий | Порог |
|---|---|
| Component eval: JSON parse rate Scout | ≥ 95% |
| Component eval: precision нормализации характеристик | ≥ 0.75 |
| Component eval: полнота секций PRD | 100% (все обязательные поля) |
| System eval: task success rate на 10 эталонных запросах | ≥ 80% |
| RAG: Recall@5 на размеченных запросах к KB | ≥ 0.7 |
| LLM-as-judge: средняя оценка отчёта (1-5) | ≥ 3.5 |
| E2E latency p95 | ≤ 5 минут |

Подробнее в `docs/EVALS.md`.

## 7. Out of scope

Система **не делает**:
- Не публикует и не редактирует карточки на WB.
- Не закупает товар и не взаимодействует с поставщиками.
- Не считает финансовую модель целиком — только базовую юнит-экономику через инструмент.
- Не учитывает сезонность за пределами того, что видно в текущих отзывах (нет исторических данных).
- Не работает с другими маркетплейсами (Ozon, Я.Маркет) — только WB.
- Не отслеживает изменения в реальном времени — каждый запуск независим.
- Не делает выводов о юридической стороне выхода в нишу (только список обязательных документов из KB).

## 8. Допущения и риски

**Допущения:**
- `search.wb.ru` будет доступен на момент демонстрации (если нет — есть mock-режим).
- Пользователь умеет ставить Docker и Ollama (инструкция в README).
- Модели Qwen2.5 / Llama 3.2 продолжают быть доступными в Ollama Library.

**Риски:**
| Риск | Митигация |
|---|---|
| WB меняет формат `search.wb.ru` | Mock-режим как fallback; явное предупреждение в README |
| 3B-модель нестабильно выдаёт JSON | Few-shot в промптах + Pydantic-валидация + retry; fallback на Llama 3.2 |
| OOM на 4 ГБ VRAM при одновременной работе LLM+эмбеддер | Эмбеддер вынесен на CPU, реранкер опционален |
| Длинные карточки не помещаются в контекст 4096 | Truncation по приоритету: title > attrs > description |
